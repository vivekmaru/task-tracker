run, observe, recover

The runtime is small; the sequence matters.

PostgreSQL is required. Server and worker are separate long-lived processes. The reverse proxy owns TLS, and migrations are applied deliberately before process replacement.

Local development

go run ./cmd/forge migrate
go test ./...
go run ./cmd/forge server --config forge.local.json

Use the README smoke path for workspace/project creation, claim/checkpoint/complete, proof storage, and web inspection.

Production deployment

cd deploy/compose
docker compose config
docker compose up -d
curl -fsSk https://localhost/livez

Compose provides Postgres, one-shot migration, server, worker, and Caddy. Systemd is the alternate bare-metal route.

operator checks

Use the right signal for the failure.

Process alive
GET /livez

Confirms the server is running; it does not query dependencies.

Database ready
GET /readyz

Pings the configured PostgreSQL dependency without exposing its connection string.

Work stalled
forge worker --once

Runs maintenance and webhook delivery deterministically; normal operation uses the long-running worker.

Webhook trouble
Inspect delivery guidance

Deliveries are durable, policy constrained, and retried through the worker rather than during a request.

Release regression
Follow upgrade and recovery guidance

Back up first; migrate forward; verify readiness and a lifecycle smoke flow. Never run down migrations in production.

Authentication

admin_token authenticates API callers and signs stateless web cookies. Rotate it to revoke outstanding sessions and update API clients together.

Artifacts and logs

Proof bytes live in a configured local root or S3-compatible bucket; incident output must never disclose database URLs, tokens, webhook secrets, query strings, or artifact contents.
